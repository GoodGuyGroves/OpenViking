# OpenViking

Local context database (RAG knowledge base) for the Oaasis workspace. Stores and semantically retrieves research, documentation, and architectural knowledge using OpenAI embeddings.

## Quick Reference

| Component | Port | Description |
|---|---|---|
| nginx proxy | 1933 | Unified entry point, routes to instances |
| work instance | 1940 | Work/Oaasis knowledge base |
| personal instance | 1941 | Personal knowledge base |
| MCP server | 2033 | Multi-instance MCP endpoint |
| deep-researcher | 8001 | Research pipeline (lives in `../deep-researcher/`) |

## Running

```bash
# Enter dev shell (provides python3, uv, nginx, git)
nix develop

# Install/sync Python deps
uv sync

# Start everything (instances + nginx + MCP server + deep-researcher)
ov-manager start

# Check status
ov-manager status

# Stop everything
ov-manager stop
```

Selective start/stop:
```bash
ov-manager start work          # single instance
ov-manager start services      # only MCP + deep-researcher
ov-manager restart personal    # restart one instance
```

## Architecture

```
                    ┌─────────────────────────────┐
                    │  MCP Server (:2033)          │
                    │  mcp-server.py               │
                    │  /mcp/work, /mcp/personal    │
                    └──────────┬──────────────────┘
                               │ HTTP
                    ┌──────────▼──────────────────┐
                    │  nginx proxy (:1933)         │
                    │  /work/ → :1940              │
                    │  /personal/ → :1941          │
                    └──┬──────────────────────┬───┘
                       │                      │
          ┌────────────▼───────┐  ┌───────────▼────────┐
          │ openviking-server  │  │ openviking-server   │
          │ work (:1940)       │  │ personal (:1941)    │
          │ data/work/         │  │ data/personal/      │
          └────────────────────┘  └─────────────────────┘
```

The MCP server is a single process that creates one FastMCP endpoint per instance, each mounted at `/mcp/<name>`. Tool calls are routed through the nginx proxy to the correct backend.

## Key Files

| File | Purpose |
|---|---|
| `instances.json` | Defines instances, ports, services, and proxy port |
| `ov-manager` | Bash script for lifecycle management (start/stop/restart/status) |
| `mcp-server.py` | Custom multi-instance MCP server (FastMCP + Starlette) |
| `ov-work.conf` | OpenViking config for the work instance |
| `ov-personal.conf` | OpenViking config for the personal instance |
| `ov.conf` | Default config (used by CLI when `OPENVIKING_CONFIG_FILE` is set via `.envrc`) |
| `flake.nix` | Nix dev shell (python3, uv, nginx, git) |
| `.envrc` | direnv setup: activates flake, syncs venv, loads `.env`, sets `OPENVIKING_CONFIG_FILE` |
| `nginx.conf` | Auto-generated by ov-manager (gitignored) |

## Configuration

**Instance configuration** lives in `instances.json`. Each instance needs:
- A unique port
- A config file (`ov-<name>.conf`) pointing to its data directory
- A `data` field (used by ov-manager for log file placement)

**OpenViking config** (`ov-*.conf`) defines:
- `storage.workspace` - path to the instance's data directory
- `embedding.dense` - OpenAI embedding model and dimension (text-embedding-3-large, 3072-dim)
- `vlm` - Vision/language model for generating summaries (gpt-5.4)

**Secrets**: `OPENAI_API_KEY` must be in `.env` (gitignored). Config files reference it as `$OPENAI_API_KEY`.

## Data Model

Each instance stores data in its own directory (`data/work/`, `data/personal/`):

```
data/<instance>/
  _system/queue/     # Processing queue
  temp/upload/       # Upload staging
  vectordb/context/  # Milvus vector embeddings (3072-dim, flat index)
  viking/default/    # Content filesystem
    resources/       # Ingested documents and research
    agent/           # Agent data
    session/         # Session data
    user/            # User data
```

### Tiered Content (L0/L1/L2)

Resources are stored with three levels of detail, auto-generated by the VLM:
- **L0 (Abstract)** - one-line summary, cheapest to retrieve
- **L1 (Overview)** - multi-paragraph structured summary
- **L2 (Full Content)** - complete document text

## MCP Tools

The MCP server exposes 12 tools per instance:

**Search**: `search`, `grep_content`, `find_by_pattern`
**Read**: `read_content`, `get_overview`, `get_abstract`
**Navigate**: `list_contents`, `get_resource_info`, `get_relations`
**Manage**: `add_resource`, `add_memory`, `health_check`

## Adding a New Instance

1. Add the instance to `instances.json` with a unique port
2. Create `ov-<name>.conf` pointing `storage.workspace` to `./data/<name>`
3. Run `ov-manager restart` - nginx config is regenerated automatically, MCP server picks up the new instance

## Development Notes

- `nginx.conf` is auto-generated by `ov-manager` from `instances.json` - do not edit manually
- Process PID files and logs live in `run/` (gitignored)
- Instance server logs go to `data/<instance>/server.log`
- The MCP server uses stateless HTTP mode with `streamable_http_path="/"`
- ov-manager sources both `.env` and `../deep-researcher/.env` when starting services
